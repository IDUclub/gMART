# Urban Data QA agent (внешний, сгруппированный Urban MCP)

Агент `urban-data-qa` отвечает на вопросы по **урбанистическим данным**, вызывая инструменты
внешнего, сгруппированного **Urban MCP** (`URBAN_DATA_MCP_SERVER` — базовый URL, например
`https://urban-mcp.example.ru/mcp`). Сервер разбит на группы, каждая — на своём под-пути; агент
подключается **ко всем группам сразу** и агрегирует их инструменты в один список:

| Группа | Путь | Что даёт |
| --- | --- | --- |
| `dictionaries` | `/mcp/dictionaries` | справочники: типы объектов/сервисов/территорий, единицы измерения и т.п. |
| `territories` | `/mcp/territories` | территории и всё, что к ним привязано (сервисы, физ. объекты, нормативы, буферы, функциональные зоны). |
| `indicators` | `/mcp/indicators` | индикаторы по территориям и сценариям. |
| `physical_objects` | `/mcp/physical_objects` | физические объекты по id. |
| `projects` | `/mcp/projects` | проекты и сценарии (в т.ч. приватные). |
| `soc_groups` | `/mcp/soc-groups` | социальные группы и ценности. |

Только группа `projects` авторизована — токен пользователя (Urban API JWT) пробрасывается только
в её MCP-клиент (нужен для приватных проектов); остальные пять групп — открытые
справочно-пространственные данные, без токена. Из инструментов `projects` исключён `CreateProject`
— единственная мутирующая операция во всём каталоге: агент строго read-only.

В отличие от остальных агентов, набор инструментов не зафиксирован в коде — агент получает его
динамически при старте запроса (по всем группам параллельно; недоступная группа просто
пропускается, не роняя весь запрос) и сам решает, какие инструменты и с какими аргументами
вызывать.

Общие механики — формат SSE, переподключение по `request_id` — те же, что в
[frontend-service.md](frontend-service.md). Ниже только специфика этого агента.

## Как работает

Нативный tool-calling цикл (до 5 раундов):

1. **tools_loading** — агент параллельно опрашивает все 6 групп и агрегирует их инструменты.
2. **executing** — модель сама решает, какой инструмент вызвать и с какими аргументами; так может
   повторяться несколько раз подряд (например, сначала найти территорию, потом её сервисы).
   Модель знает текущий `scenario_id` запроса (если он передан) и умеет подставлять его в
   инструменты, которым он нужен (`scenario_id`/`project_id` — обычные обязательные параметры
   схемы конкретного инструмента, не скрытые протокольные поля).
3. **answer_drafting** — как только модель решает, что данных достаточно (или закончились раунды),
   формируется и стримится финальный ответ на основе полученных данных.

Если инструмент вернул пространственные данные (`FeatureCollection`, на любом уровне вложенности
ответа), она отправляется клиенту отдельным событием `feature_collection`, чтобы её можно было
показать как слой на карте — независимо от того, что именно упомянуто в тексте ответа.

При истечении срока действия токена (актуально для вызовов группы `projects`) используется тот же
механизм `token_expired` / `POST /pipelines/{request_id}/token`, что и у restriction/provision
агентов.

> Реконнект (`request_id`) после **завершения** пайплайна лишь повторяет буферизованные события.
> Реконнект **посреди** цикла (редкий случай — цикл короткий) перезапускает tool-calling с нуля, а
> не восстанавливает точное состояние диалога с моделью — вызовы инструментов дёшевы и ограничены
> 5 раундами, так что это не создаёт заметной задержки.

## Эндпоинты

### `GET /urban-data/qa/stream` — SSE

```http
Authorization: Bearer <access_token>
```

| Параметр | Тип | По умолч. | Описание |
| --- | --- | --- | --- |
| `request` | `string` | обязательный | Вопрос пользователя. |
| `model` | `string` | `gpt-oss:20b` | Имя модели Ollama. |
| `temperature` | `number` | `1.0` | Температура генерации. |
| `scenario_id` | `number \| null` | `null` | ID сценария Urban API. Если `chat_id` не передан — создаётся чат с этим `scenario_id` и выведенным из него `project_id`. Также подставляется моделью в scenario/project-scoped инструменты. |
| `chat_id` | `string \| null` | `null` | UUID чата из Chat Storage для продолжения диалога. |
| `request_id` | `string \| null` | `null` | UUID пайплайна для переподключения после разрыва. |

Эндпоинт возвращает ошибку, если `URBAN_DATA_MCP_SERVER` не сконфигурирован на бэкенде.

### `POST /urban-data/a2a` — A2A JSON-RPC

Карточка агента: `GET /urban-data/.well-known/agent-card.json` (имя `urban-data-qa-agent`).
Методы: `SendMessage`, `SendStreamingMessage`, `GetTask`, `ListTasks`, `CancelTask`,
`GetExtendedAgentCard`. `scenario_id` / `chat_id` передаются в `params.message.metadata`
(оба необязательны). A2A-запуски не пишут историю в ChatStorage (`persist_history=False`).

## Типы SSE-событий

Конверт стандартный (`{ "type", "content" }`). Набор событий этого агента:

| Тип | Назначение |
| --- | --- |
| `pipeline_started` | первое событие; сохранить `content.request_id`. |
| `status` | прогресс; `content.status` ∈ `tools_loading`, `executing`, `answer_drafting`. |
| `tool_call` | вызов инструмента Urban MCP (`content.mcp_source = "URBAN_DATA_MCP_URL"`). |
| `feature_collection` | слой из результата инструмента; `content = { name, feature_collection }`. |
| `chunk` | текст финального ответа; `content = { text, done }`. |
| `service_event` | создан чат (`chat_created` с `chat_id` / `chat_title`). |
| `warning` | неблокирующее предупреждение (см. `frontend-norms-qa.md` — тот же формат). |
| `token_expired` / `pipeline_suspended` | истёк токен группы `projects` — см. `frontend-service.md`. |
| `error` | ошибка; следом приходит завершающий `chunk` с `done: true`. |

В A2A-варианте те же события маппятся в артефакты/статусы задачи: `chunk` → текстовый артефакт
(`urban-data-agent-text`, накопительный), `feature_collection` → GeoJSON-артефакт
(`geojson-<имя-слоя>`), `token_expired` → промежуточный статус `working`, `pipeline_suspended`/
`error` → финальный статус `failed`; `tool_call`/`service_event`/`pipeline_started` — внутренние,
наружу не отдаются.
